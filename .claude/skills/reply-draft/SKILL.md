# reply-draft — 회신 절차 (공통 뼈대)

상대 원문 + 맥락을 받아 **회신 초안 1편을 생성한다**. 이 스킬은 절차(엔진)만 안다 — 말투·금지 표현·첨부 같은 **내 값은 전부 내 세계 사전에 산다**.

**읽는 파일 (판단 재료는 전부 파일에 — 이 스킬 본문에 개인 데이터 없음):**

```bash
WS_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
```

- **사전** `$WS_ROOT/00-system/08-registry/내-세계-사전.yaml` `회신-원칙` 절 — 금지 표현·톤·첨부 경로·서명 (비어 있으면 아래 "기본 원칙"만 적용)
- **선언표** `$WS_ROOT/00-system/08-registry/자리-선언표.yaml` — 진행표 표면(있으면 발송 후 추적 대상)

## 절대 규칙 (엔진 — 누구에게나 같음)

1. **확인 안 된 것 만들지 않기** — 워크스페이스에서 직접 확인한 사실만 쓴다. 유입 경로·소개자·상품 속성을 원문에 없는 형태로 창작하지 않는다. 원문 표현은 원문 그대로 인용.
2. **발송은 사람 몫** — 이 스킬은 실물 초안까지 차린다. 발송·전송·확약은 사용자가 "보내줘"라고 명시한 뒤에만 (실행 수위 ③: 밖으로 나가는 것).
3. **초안은 항상 전문(full text) 출력** — 요약·발췌로 대체 금지. 톤·단어를 검토하려면 전문이 필요하다.
4. **초안 피드백은 재작성** — 부분 땜질로 단어만 바꾸지 않는다. 피드백을 받으면 Step 2 맥락 판단부터 다시 지나 전 검증을 재통과한다.

## Step 1: 입력 확인

- 상대 원문 / 채널(메일·DM·문자) / 스레드 단계(첫 회신·후속·거절·확정 후)를 파악
- 원문에 없는 정보는 만들지 않는다. 원문 자체가 없거나 회신 의도(수락/거절/조건부)가 모호하면 **여기서 멈추고 질문** — 그 외에는 추정 후 진행(검증이 안전망)

## Step 2: 맥락 판단

- **상대 관점 먼저**: 표면 질문 뒤의 실제 고민 / 상대의 비용·일정 제약
- **유사 사례**: 진행표·과거 회신에서 비슷한 상황을 grep — 검색 없이 작성 금지
- **일정**: 일정을 언급해야 하면 캘린더·진행표를 실제로 확인한 값만 (뭉뚱그린 "다음 달쯤" 금지)

## Step 3: 작성 — 구성 골격 (positive rule)

> **인사 1문장 + 호응 ≤1문장 + 용건(사실·경로 안내) + 닫기 1문장**

- **호응은 ≤1문장** — 상대가 쓴 표현 하나를 짧게 받는 것까지. 상대 니즈 전체를 내 문장으로 재서술(echo)하지 않는다
- **근거 원칙** — 본문 모든 문장의 근거는 (a) 내가 실제 한 말 (b) 내 문서의 사실 (c) 확인된 사실 — 셋 중 하나. 셋 다 아닌 일반론·공감 논증은 못 들어간다
- **빈 약속 금지** — 지키지 못할 구체적 약속·후속 유도("기회 되면", "언제든지") 대신 단정하게 닫는다
- 사전 `회신-원칙` 절의 톤·표현 규칙을 적용

## Step 4: 검증

1. 사전 `회신-원칙.금지-표현`의 각 패턴을 초안 본문에 grep — 매치 = FAIL → 해당 문장 재작성 후 재검증
2. 첨부를 언급했으면 사전 `회신-원칙.첨부.기본-경로`에서 `ls`로 실존 확인 — 없는 파일을 "첨부합니다"라고 쓰지 않는다
3. 문장별 근거 태깅(위 3출처) — 어디에도 못 붙는 문장은 삭제

## Step 5: 출력

1. 판단 요약 (한 단락)
2. **회신 초안 전문** (인사부터 서명까지 그대로)
3. 확인 필요 사항 (사용자에게 / 상대에게)
4. 검증 결과 (PASS/FAIL 표)
5. 발송 안내 한 줄: "확인 후 직접 발송하시거나 '보내줘'라고 말씀 주세요"

## Step 6: 발송 후 추적

발송이 확인되면(사용자 발송 알림 포함), 선언표에 진행표 표면이 선언돼 있으면 해당 행 갱신을 **제안**한다 (표면 변경 = 실행 수위 ②: 제안 → 승인).
