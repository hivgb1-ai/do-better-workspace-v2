# Do Better Drive · 6단계 — 내보내기 게이트

> Do Better Drive 7단계 중 6단계. 5단계(drive-review)가 검토를 통과시켰다면,
> 이 단계는 그걸 **실제로 세상에 내보낸다** — 발행·출간·모객·전달·판매.
> 개발자용 gstack의 ship을 비개발 프로젝트에 맞춘 것.

## 이 스킬의 역할

검토 통과한 산출물을 **계획이 정한 유통 경로로 실제로 내보낸다.** 아무리 잘 만들어도 안 내보내면 아무도 못 본다 — 이 단계가 마지막으로 그걸 실제로 내보낸다. 하지만 내보내기는 대개 **되돌리기 어렵다**(공개 발행·발송·출간·계약은 한 번 나가면 남의 손·저장된 사본에 남는다). 그래서 나가기 전에 마지막 확인 게이트를 둔다.

**절대 규칙: 발행·발송을 여기서 재구현하지 않는다.** 실제 내보내기는 이 워크스페이스·프로필의 기존 스킬(ghost-publish·gmail·slack·naver-commerce·note-publish 등)에 위임한다. 여기서 발송 도구를 새로 짜지 않는다.
**검토 안 통과면 안 내보낸다.** 5단계 판정이 합격이어야 한다.
**밖으로 나가는 건 사람이 확정한다.** 되돌리기 어려운 발신은 초안·설정까지 다 차린 뒤 멈추고, 사람이 "내보내"라고 해야 나간다.

## 작동

### 0. 입력 읽기 + 준비 확인
- 5단계가 갱신한 `progress.md`의 **"검토"**(판정)와 **"계획"**(유통 경로)과 **"생산"**(생산물 위치)을 읽는다.
- **검토 판정이 합격이 아니면** → drive-review(5단계)로 돌려보낸다. 안 통과한 걸 내보내지 않는다.
- **조건부 통과였으면** — 남기기로 한 작은 수정이 실제로 됐는지 먼저 확인한다. 안 됐으면 그것부터 처리한다.
- `CLAUDE.md` 프로필 · `08-registry` (있으면 참고).

### 1. 나갈 준비 됐나 (마지막 점검)
- 검토 통과를 확인했다. 생산물이 실제로 그 자리에 있나(경로를 연다). 계획에 유통 경로가 적혀 있나.
- 이 셋 중 하나라도 비면 억지로 진행하지 않는다 — 빈 것을 채우거나 앞 단계로 돌린다.
- **검토받은 그 버전이 맞나** — 5단계 검토 뒤에 산출물을 손댔다면, 지금 내보내려는 게 검토를 통과한 그 버전인지 확인한다. 사소한 손질이면 바뀐 데만 다시 본 뒤 내보내고, **실질적으로 바뀌었으면 5단계로 되돌려** 다시 검토받는다 (검토 안 된 버전이 되돌리기 어렵게 나가는 걸 막는다).

### 2. 유통 경로 확인 (계획대로)
계획이 정한 채널로 나가는지, 그 채널이 실제로 준비됐는지 본다.
- 어디로 나가나 (발행처·받는 사람·판매 페이지 등)와 그게 준비됐나(계정·주소·페이지가 실제로 있나).
- **경로가 없거나 안 준비됐으면** 억지로 내보내지 말고 한 번 묻는다: 지금 준비할까 / 다른 채널로 갈까 / 미룰까. (계획에 유통이 빠졌던 거면 그 자체가 3단계가 놓친 것 — 짚어준다.)

### 3. 내보내기 전 마지막 점검 (되돌리기 어려울수록 깐깐히)
- **비가역 확인** — 공개 발행·발송·출간·계약은 되돌리기 어렵다. 나가면 남의 손·검색·복사본에 남아 지워도 흔적이 남는다. 되돌리기 쉬운 것(초안 저장)과 어려운 것(공개 발행)을 가르고, 어려운 것은 한 번 더 본다.
- **민감정보 점검 (이림 규칙 — 반드시)** — 밖으로 나가는 것에 노출되면 안 되는 게 없나 훑는다:
  - 공개 발행물(SNS·블로그·뉴스레터 등)에 **매출·계약금액·견적가·단가**가 들어갔나 → 빼거나 정량 우회(건수·인원·회차)로 바꾼다.
  - **회사명 + 금액을 동시에** 노출하나 → 특히 뺀다.
  - API 키·비밀번호·개인정보가 섞였나 → 뺀다.
  - (인바운드 회신이면 별도 규칙 — 상대가 가격을 안 물었으면 본문에 넣지 말고 첨부로. 이건 inbound-reply가 전담하니 그쪽 관례를 따른다.)
- **받는 사람·채널·때 확인** — 누구에게, 어느 채널로, 언제 나가나. 잘못된 수신자·엉뚱한 채널이 아닌가 확인한다.
- **작게 먼저 보내 확인 (채널이 되면)** — 전면 발신 전에 작게 먼저 내보내 수신자가 보는 그대로 연다: 자기에게 테스트 발송·비공개 미리보기·소수 선전송. 오타·링크·깨짐을 실제 형태로 본 뒤 전면 발신한다. 확정 게이트보다 강한 안전장치다. **처음 쓰는 채널이면** 특히 — 무엇이·어디로·어떻게 나가는지 한 번 되짚고 채널이 실제로 동작하는지 본 뒤 진행한다(다음부터는 건너뛴다).

### 4. 확정 게이트 (사람이 "내보내"라고 해야 나간다)
- **되돌리기 어려운 발신**은 초안·설정까지 다 차린 뒤 **멈추고** 사람에게 확인받는다. 무엇을 · 어디로 · 언제 내보내는지 요약해 되읽어주고(`AskUserQuestion` 또는 "이대로 내보낼까요?"), 사람이 확정한다.
- **되돌리기 쉬운 것**(초안 저장·비공개 등록 등)은 실행하고 보고하면 된다 — 매번 묻지 않는다.

### 5. 내보내기 (기존 스킬에 위임)
확정된 것을 계획의 채널로 **실제로 내보낸다.** 맞는 발행·발송 스킬을 부른다 — 여기서 재구현하지 않는다.
- 채널이 여럿이면(한 콘텐츠를 여러 채널로) 채널마다 내보낸다. 각 채널에 맞는 형식 변환도 그 채널 스킬에 맡긴다(여기서 손으로 바꾸지 않는다).
- **재실행이면 이미 나간 채널은 다시 안 보낸다** — 여러 채널 중 일부만 실패해 다시 돌릴 때, 이미 나간 채널을 또 보내면 중복 발송(뉴스레터 두 번·중복 게시)이 된다 — 되돌릴 수 없다. 채널마다 "이미 나갔나"를 먼저 확인하고 나간 건 건너뛴다 (확인·기록은 다시 하되 발송만 건너뛴다).
- **맞는 스킬이 없으면** 여기서 최소로 처리하되, 새 발송 스킬을 짜느라 멈추지 않는다.

### 6. 내보낸 것 확인 + 기록
- **실제로 나갔나 + 의도대로 보이나 확인한다** — 발행 주소가 열리나·발송이 완료됐나에서 그치지 않고, **수신자가 보는 그대로 열어** 내용이 의도대로 나오나 본다(본문·이미지·링크가 안 깨졌나). 주소가 열리는 것과 제대로 보이는 것은 다르다. "나갔을 것"으로 넘기지 않고 확인한 근거를 든다.
- **안 나갔으면** 왜인지 보고 다시 시도하거나, 막히면 무엇이 막혔는지 적고 멈춘다.
- **이미 나갔는데 잘못됐으면** 되돌리기가 가능한지 본다(발행 취소·비공개 전환·정정 공지·회수). 가능한 최소 조치를 사람에게 제안하고, 완전히 못 되돌리면 무엇이 남는지 적어 알린다. (대전제가 "되돌리기 어렵다"인 만큼, 나간 뒤의 최소 대응까지 챙긴다.)

### 7. 핸드오프 (산출)
`progress.md`를 갱신한다:
- **"지금 상태"** 에 내보낸 것 기록 — 무엇을 · 어느 채널로 · 어디에(주소·위치) · 언제
- **"다음 한 걸음"** 갱신
- frontmatter `status: 발행`, `updated` 갱신
- 로그에 `- <YYYY-MM-DD> 내보냄 (drive-ship): <채널·위치>` append
- 안내: **"내보냈어요. 다음은 7단계 — 돌아보기·배우기."**

여러 산출물을 내보내는 프로젝트면 산출물·채널마다 이 절차를 다시 돈다.

## 규칙 요약
- **재구현 금지** — 발행·발송은 기존 스킬에 위임. 여기서 발송 도구를 새로 짜지 않는다.
- **검토 통과가 조건** — 합격 아니면 안 내보낸다. 5단계로 되돌린다.
- **유통은 계획대로** — 계획이 정한 채널로. 경로가 없거나 안 준비됐으면 묻는다.
- **비가역은 깐깐히** — 되돌리기 어려운 발신은 한 번 더 본다. 나간 뒤 잘못됐으면 되돌리기 가능한지 보고 최소 조치를 제안한다.
- **작게 먼저, 재실행은 중복 금지** — 전면 발신 전 테스트 발송·미리보기로 확인. 재실행 시 이미 나간 채널은 다시 안 보낸다.
- **민감정보 점검은 필수** — 공개 발행물에 금액·단가·회사명+금액·키를 노출하지 않는다. 정량 우회로 바꾼다.
- **사람이 확정** — 되돌리기 어려운 건 초안까지 차려 멈추고, 사람이 "내보내" 해야 나간다.
- **나갔는지 확인** — "나갔을 것"으로 넘기지 않는다. 확인 근거를 든다.
