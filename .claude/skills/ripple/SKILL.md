파일 수정 후 연관 파일을 탐색하여 누락된 업데이트를 찾고 함께 수정합니다.

**수행할 작업:**

## 1. 변경 파악

이 스킬은 인라인으로 실행되므로, 이 세션에서 Write/Edit한 파일을 이미 알고 있다.
별도의 파싱이나 git status 확인 없이, 이 세션에서 수정한 파일 목록을 바로 정리한다.

변경 파일에서 키워드 추출:
- 파일명 (예: `daily-note/SKILL.md`)
- 스킬/에이전트/프로젝트 이름 (예: `daily-note`, `11-website-redesign`)
- 변경 종류 (신규/수정/삭제/이름변경)

## 2. 연관 파일 탐색

워크스페이스 루트는 git root에서 자동 감지. git repo가 아니면 현재 작업 디렉토리(워크스페이스 루트)를 사용한다:

```bash
git rev-parse --show-toplevel 2>/dev/null || pwd
```

**탐색 제외:**
- `90-archive/`
- `.git/`
- `.venv/`, `node_modules/`, `venv/` 등 생성물

### A. 직접 참조 (Grep)

워크스페이스 전체에서 변경된 파일명/스킬명을 참조하는 파일 검색.

```
Grep: "daily-note" -> 참조하는 .md 파일 목록
```

### B. 구조적 연관 (Glob + 패턴)

변경 유형에 따른 구조적 연관 파일:
- 스킬 수정/신규 -> `.claude/skills/README.md` (스킬 카탈로그에 등재됐는지 확인)
- 에이전트 수정/신규 -> `.claude/agents/README.md` (에이전트 카탈로그에 등재됐는지 확인)
- 프로젝트 파일 수정 -> 같은 프로젝트 폴더 또는 상위 폴더에서 `progress.md` Glob (아래 D 참조)

### C. 시간 기반 (오늘 daily note)

오늘의 daily note 파일이 이미 존재하면 다음 섹션을 업데이트 대상으로 포함:
- "완료한 일" 섹션: 이번 세션의 주요 성과에 기록할 내용이 있으면 추가

경로: `40-personal/41-daily/YYYY-MM/YYYY-MM-DD.md`

오늘 파일이 없으면 이 경로는 건너뛴다 (새로 생성하지 않음).

### D. 진행 추적 (progress)

변경된 파일이 `10-projects/XX-name/` 프로젝트 폴더 아래인지 확인하고, 그 프로젝트의 `progress.md`를 탐색한다.

```
Glob: "progress.md" (변경 파일의 상위 프로젝트 폴더 내)
```

- 하위 폴더(`artifacts/` 등)에서 변경이 일어났으면 **프로젝트 루트 폴더**(`10-projects/XX-name/` 또는 `10-projects/XX.YY-name/`)까지 올라가며 탐색한다.
  예: `10-projects/11-website-redesign/artifacts/` → `10-projects/11-website-redesign/`
- **발견되면**: 업데이트 대상에 추가 (타임라인 행 추가 + "최근 업데이트" 날짜 갱신).
- **프로젝트 루트까지 탐색해도 없으면**: 리포트에 "progress.md 없음 - 생성할까요?" 항목 추가.
  - 생성 위치는 **프로젝트 루트 폴더에 한정** (`10-projects/XX-name/progress.md`). `artifacts/` 등 하위 폴더에는 만들지 않는다.
  - 이름은 항상 plain `progress.md`.

## 3. 영향도 리포트

탐색 결과를 테이블로 정리하여 사용자에게 제시:

```
| # | 파일 | 연관 유형 | 현재 상태 | 필요 조치 |
|---|------|-----------|-----------|-----------|
| 1 | .claude/skills/README.md | 구조적 | ripple 미등재 | 카탈로그에 추가 |
| 2 | 10-projects/11-website-redesign/progress.md | 진행 추적 | 파일 없음 | 템플릿으로 생성 |
| 3 | 2026-06-20.md | 시간 기반 (완료한 일) | 성과 미기록 | 주요 성과에 추가 |
```

- "필요 조치 없음" 파일은 리포트에서 **제외**
- AskUserQuestion으로 어떤 것을 업데이트할지 확인 (전체/선택)

## 4. 업데이트 실행

사용자가 승인한 파일만 업데이트:
- 각 파일을 Read 후 적절한 위치에 Edit
- 변경 내용에 맞는 **최소한의 수정** (과잉 수정 금지)
- 업데이트 완료 후 결과 요약

### progress.md 생성 (승인된 경우)

`00-system/01-templates/progress-template.md`를 읽어 다음을 채워 `10-projects/XX-name/progress.md`로 Write:
- `[프로젝트명]` → 프로젝트 폴더명
- `시작`/`최근 업데이트` 날짜 → 오늘 (`date +%Y-%m-%d`)
- 타임라인 첫 행 → 이번 세션 변경 요약 (날짜·한 일)

템플릿 파일이 없으면 아래 임베드 템플릿을 사용한다 (clone이 불완전해도 동작):

```markdown
# [프로젝트명] — 진행 로그

> 시작: YYYY-MM-DD · 최근 업데이트: YYYY-MM-DD

## 타임라인

| 날짜 | 한 일 | 결과 / 메모 |
|------|-------|-------------|
| YYYY-MM-DD | 프로젝트 시작 | |

## 다음 액션

- [ ]

## 메모

-
```

### progress.md 갱신 (이미 있는 경우)

재생성하지 않는다. 타임라인 표에 이번 세션 행을 **추가**하고, 상단 "최근 업데이트" 날짜를 오늘로 바꾼다.

**참고사항:**
- "빠짐없이"가 핵심 - 하나라도 놓치면 나중에 발견하기 어려움
- 과잉 수정보다 과소 수정이 나음 - 확실한 것만 제안
- daily note 업데이트는 오늘 날짜 파일이 이미 있을 때만
- **daily note "완료한 일" 체크**: 프로젝트와 무관한 작업(스킬 생성, 시스템 개선, 내부 도구 등)도 "완료한 일"에 기록할 가치가 있으면 추가
- progress.md는 단계 status 같은 고정 구조를 강제하지 않는다. 진행 로그를 계속 쌓다가, status 칸이나 상위 마스터 progress가 필요해지면 그때 덧붙인다. walk-up 탐색은 상위 폴더의 progress.md도 찾으므로 나중에 마스터를 두어도 자동으로 잡힌다.

## 5. Wiki Ingest 제안

업데이트 실행 완료 후, 위키 축적 가능성을 체크하고 **제안만** 한다.
wiki-ingest는 별도 스킬이므로 ripple이 직접 실행하지 않는다.

**절차:**
1. 이 세션에서 작업한 내용에서 핵심 토픽/키워드 추출
2. `30-knowledge/00-wiki/index.md` 읽어서 기존 토픽 목록 확인
3. 관련 기존 토픽이 있거나 새 토픽 가치가 있으면 구체적으로 제안
4. 사용자가 승인하면 **`wiki-ingest` 스킬 호출을 안내**한다 (ripple이 직접 wiki를 수정하지 않음)
5. 축적할 게 없으면 **조용히 종료** (질문하지 않음)

**원칙:**
- ripple은 제안까지만. wiki 수정은 wiki-ingest 스킬의 책임
- 축적할 가치 없으면 질문 없이 종료
