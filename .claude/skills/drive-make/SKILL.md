# Do Better Drive · 4단계 — 생산 게이트

> Do Better Drive 7단계 중 4단계. 3단계(drive-plan)가 "어떤 방식으로·무엇이 되면 합격인가"를 잠갔다면,
> 이 단계는 **처음으로 실제로 만들게 지휘한다.** 단 통짜 하나로 직진하지 않는다.
> 개발자용 gstack의 design-shotgun(변형 여러 개 비교)+design-html(확정본 풀생산)을 비개발 프로젝트에 맞춘 것.

## 이 스킬의 역할

Do Better Drive에서 **처음으로 만들기가 허용되는 단계.** (앞 3개는 만들기 금지였다.) 하지만 이 스킬은 스스로 산출물을 만들어내는 도구가 아니라, **생산을 제대로 된 방향으로 지휘하는 감독 게이트**다. 완성본 하나로 곧장 가지 않는다 — 서로 다른 변형을 여러 개 뽑아 비교하고, 취향으로 하나를 확정한 뒤, **그 방향만** 기존 스킬로 풀생산시킨다. 시작부터 한 방향에 갇혀 엉뚱한 걸 열심히 완성하는 걸 막는다.

**핵심 순서: 컨셉으로 여러 개 → 확정 게이트 → 가벼운 실물로 비교 → 취향 택1 → 풀생산.** 건너뛰지 않는다.
**재구현 금지: 풀생산은 이미 있는 콘텐츠·산출 스킬을 부른다.** 여기서 생산 도구를 새로 짜지 않는다.

## 작동

### 0. 입력 읽기 + 이전 생산 확인
- 3단계가 갱신한 `progress.md`의 **"계획"**(고른 접근·구조·순서·층 구분)과 **"합격 기준"**을 읽는다.
- **둘 중 하나라도 없으면** 아직 계획이 안 잠긴 것 → drive-plan(3단계)로 돌려보낸다. 여기서 즉석으로 계획 잡지 않는다.
- **이 산출물을 전에 만든 적 있나** — 로그·"생산" 칸을 보고, 이미 변형을 뽑았거나 생산물이 있으면 "새로 뽑기 vs 그때 것 이어서 다듬기"를 먼저 묻는다.
- `CLAUDE.md` 프로필(특히 "판단 성향") · `08-registry`(있으면 참고).

### 1. 무엇을 만들지 정리 (무엇을 · 어떤 스킬로)
만들 산출물을 정리한다. **계획에 이미 있는 건 채워 넣고, 빈 것만 묻는다** (5가지):
1. **누구에게** — 이 산출물을 받을 사람 (프레임의 대상과 이어짐)
2. **무슨 일을 하게** — 받은 사람이 뭘 하길 바라나 (읽고 신청? 보고 설득? 배우고 실행?)
3. **이미 있는 것** — 재사용할 톤·자료·형식 (계획의 재사용에서 끌어옴)
4. **어디서 만나 어디로** — 이걸 어디서 접하고 그다음 어디로 가나 (맥락·흐름)
5. **빗나갈 경우** — 분량 넘침·톤 어긋남·핵심 빠짐·빈 경우·처음 보는 사람

그리고 **몇 개 변형을 뽑을지** 정한다 (기본 2~3개, 중요한 산출물이면 더). 묻는 건 **최대 2번까지** — 그 뒤엔 아는 것으로 진행하고 가정을 밝힌다.
그리고 이걸 **풀생산할 기존 스킬 후보**를 찾는다. 이 워크스페이스·프로필에 있는 콘텐츠·산출 스킬을 쓴다. 맞는 스킬이 없으면 직접 만들되(아래 5. 풀생산에서 처리), 새 생산 스킬을 즉석에서 짜지 않는다.
**작은 산출물**(캡션 하나 등)은 이 절차가 무겁다 — 개수를 줄이고, 아래 3·4의 고르기를 한 번으로 합쳐도 된다 (규칙은 6단계 끝 "작은 산출물" 참고).

### 2. 지난 취향 확인
변형을 뽑기 전, 이 사람이 전에 뭘 골랐는지 읽어 그쪽으로 기울인다 — 매번 처음부터 취향을 재지 않는다.
- 프로필의 "판단 성향" + 이 프로젝트 로그의 **지난 확정**(뭘 골랐고 왜 골랐는지)을 읽는다. (새 파일은 만들지 않는다 — progress.md 로그가 취향의 기록이다.)
- **어긋남 짚기** — 이번 요청이 지난 취향과 어긋나면(예: 늘 담백한 톤을 골랐는데 이번엔 화려하게) 짚어준다: "전엔 담백한 쪽을 고르셨는데 이번엔 반대네요 — 이번만 그런 거예요, 취향이 바뀐 거예요?"

### 3. 컨셉으로 여러 개 (풀생산 전) + 확정 게이트
풀생산에 들어가기 전에, **컨셉부터** 뽑는다. 힘을 크게 들이기 전에 방향을 고르는 장치다.

**3a. 컨셉 목록** — 서로 다른 변형을 1번에서 정한 수만큼(기본 2~3개) 컨셉으로 제시 (각각 이름 + 한 줄 방향).
```
세 방향으로 갈게요:
A) "이름" — 이 방향의 한 줄 설명
B) "이름" — 이 방향의 한 줄 설명
C) "이름" — 이 방향의 한 줄 설명
```
- **안 겹치게 (반드시 지킬 테스트):** 변형은 각도·톤·구조가 **여러 개** 갈릴수록 좋다. 갈리는 게 하나뿐이고 그마저 약하면(예: 톤만 다르고 각도·구조는 같으면) 사실상 같은 것 둘일 위험이 크다. 테스트 — **"두 변형의 제목을 서로 바꿔 껴도 아무도 못 알아채면, 그건 같은 것 둘이다."** 세 사람이 따로 만든 것처럼 느껴져야지, 한 사람이 조금씩 바꿔 셋 만든 게 아니다. 약한 하나는 일부러 다른 방향으로 다시 뽑는다.
- **몸으로 만드는 산출물**(요리·실물 제작·현장)은 컨셉은 여러 개라도 실물은 싸게 못 찍는다 → 변형 수를 줄이고 그 사실을 그대로 말한다.

**3b. 확정 게이트 (풀생산 전 승인)** — `AskUserQuestion`으로 묻는다: 이 컨셉들로 갈까?
- 다 좋다 → 다음(실물 비교)으로 / 일부 바꿀래(어느 것) / 더 뽑을래 / 줄일래. 바꾸면 다시 제시하고 다시 확정. **최대 2번.**

### 4. 가벼운 실물로 비교 → 취향 택1 → 되읽어 확인
확정된 컨셉을 **가벼운 실물 샘플**로 만든다 (통째 말고 일부만 — 리드 문단·슬라이드 3장·커리큘럼 큰 틀·첫 화면). 글만으로는 진짜 느낌을 모르니, 골라 쓸 만큼만 실물로 본다.
- **나란히 비교** — 표나 나란히 배치로 한자리에서 본다. 각 샘플의 강점·약점, 그리고 **합격 기준에 비춰** 어디가 세고 어디가 약한지 짚어준다. Claude가 하나로 미는 게 아니라 **고를 재료를 차려준다.**
- **취향으로 택1 (사람 몫)** — `AskUserQuestion`으로 참가자가 하나를 고른다 (또는 "A의 이 부분 + B의 이 부분"으로 섞기). 이 선택이 **사람층의 핵심 판단** — 취향·핵심 판단이라 AI가 대신 정하지 않는다.
- **안 맞으면 다시** — 셋 다 아니면, 무엇이 약한지 피드백을 받아 다른 방향으로 다시 뽑는다(컨셉 단계로 돌아감). 단 **샘플 다시 뽑기는 1번까지** — 그 뒤엔 가장 나은 걸로 확정하고 다듬기로 넘긴다.
- **되읽어 확인** — 고른 걸 이렇게 정리해 되읽어 준다: "고른 것 = B / 좋았던 점 = … / 방향 = …. 맞아요?" 확정 전에 한 번 맞춰본다.
- 고른 이유(취향)를 로그에 한 줄 남긴다 → 다음 생산에 되먹인다(위 2. 지난 취향).

### 5. 풀생산 (기존 스킬에 위임) + 다듬기
확정된 방향 하나를 **풀버전으로 생산한다.** 1번에서 잡은 기존 콘텐츠·산출 스킬을 부른다 — 여기서 재구현하지 않는다.
- **맞는 스킬을 못 찾았으면** 여기서 확정된 방향만 직접 만든다. 단 새 생산 스킬을 짜느라 멈추지 않는다.
- 계획의 층 구분대로 완성도를 건다:
  - **AI가 하는 일** — 풀버전으로. 다루는 범위·채널·마감 다 챙긴다. 여기선 덜 하지 않는다.
  - **사람이 하는 일** — 확정된 취향·핵심 판단을 반영한다.
  - **몸으로 하는 일** — 병목이면 병목대로 잡는다 (AI 일처럼 싸다고 착각하지 않는다).
- **다듬기 루프** — 풀생산한 걸 보여주고, 고칠 데를 짚으면 반영한다. **참가자가 "됐다"고 하면 멈춘다.** 매번 처음부터 다시 만들지 말고 고칠 데만 손본다(앞 결과·참가자가 고친 것 보존). AI 티가 나는 뻔한 기본값은 피한다. 여러 번 돌아도 안 끝나면 "더 다듬을지 여기서 끝낼지" 한 번 묻는다.
- **합격 기준을 보며** 만들되, **스스로 합격이라고 확정하지 않는다** — 검사는 Do Better Drive 5단계(냉정한 검토) 몫이다.

### 6. 핸드오프 (산출)
`progress.md`를 갱신한다:
- **"생산"** 칸에 기록 — 뽑은 컨셉(무엇이 갈렸나) / 확정한 것과 이유 / 부른 스킬 / **생산물이 어디 있는지(경로·링크)**
- **"다음 한 걸음"** 갱신
- frontmatter `status: 생산`, `updated` 갱신
- 로그에 `- <YYYY-MM-DD> 생산 (drive-make): <고른 것과 이유>` append (이 한 줄이 다음 생산의 취향 근거가 된다)
- 안내: **"생산했어요. 다음은 5단계 — 냉정한 검토."**

여러 산출물을 만드는 프로젝트면(예: 콘텐츠 여러 편) 이 스킬을 산출물마다 다시 돈다. 매번 컨셉 → 확정 → 비교 → 택1 → 풀생산.
**작은 산출물**(캡션 하나 등)은 이 절차가 무거우니 3·4를 한 번으로 합친다 — 컨셉이 곧 실물이고, 확정 게이트(3b)와 취향 택1(4)을 **한 번의 고르기**로 합쳐 `AskUserQuestion`을 한 번만 띄운다. 단 "여러 개 뽑아 고른다"는 기본 순서는 유지한다.

## 규칙 요약
- **통짜 직진 금지** — 컨셉 여러 개 → 확정 게이트 → 실물 비교 → 택1 → 풀생산. 순서를 건너뛰지 않는다 (작은 산출물은 3·4를 한 번으로 합치기 허용).
- **컨셉 먼저, 풀생산은 확정 후** — 힘 크게 들이기 전에 방향을 고른다.
- **안 겹치게, 진짜로 다르게** — 제목 바꿔 껴도 모르면 같은 것. 미묘한 차이 셋은 변형이 아니다.
- **확정은 사람 몫** — 취향 택1은 `AskUserQuestion`으로 참가자가. 되읽어 한 번 맞춰본다.
- **재구현 금지** — 풀생산은 기존 콘텐츠·산출 스킬을 부른다. 없으면 직접 만들되 새 스킬을 짜느라 멈추지 않는다.
- **다시 뽑기·다듬기엔 끝이 있다** — 샘플 재생산 1번까지, 다듬기는 "됐다"에서 멈춘다.
- **층대로 완성도** — AI층 풀버전, 사람층 취향 초점, 몸노동 병목 인정.
- **스스로 합격 확정 금지** — 합격 기준을 보며 만들되, 검사는 5단계 몫.
