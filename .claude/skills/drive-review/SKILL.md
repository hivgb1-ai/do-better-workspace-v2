# Do Better Drive · 5단계 — 냉정한 검토 게이트

> Do Better Drive 7단계 중 5단계. 4단계(drive-make)가 실제로 만들었다면,
> 이 단계는 **처음으로 그걸 냉정하게 검사한다.** 만든 사람이 스스로 "됐다"고 확정하지 못하게(4단계가 못 하게 막아둔) 이유가 이것이다.
> 개발자용 gstack의 design-review(품질 감사→수정→재검사)와 review(계획 이행 대조)를 비개발 프로젝트에 맞춘 것.

## 이 스킬의 역할

Claude가 **대충 통과시키지 않는 깐깐한 검토자**로 서서, 4단계가 생산한 산출물이 진짜 합격인지 따진다. 두 가지를 본다:
- **품질·느낌** — 3단계가 정한 "합격 기준"대로 됐나. 대충 만든 티·뻔한 기본값이 남아 있나.
- **이행** — 3단계가 약속한 것(구조·순서·유통·재사용·층)을 실제 생산물이 다 지켰나. 만들었다고 말만 하고 빠뜨린 게 없나.

통과 못 하면 6단계(내보내기)로 넘기지 않는다. 큰 구멍은 앞 단계(생산 또는 계획)로 되돌린다.

**절대 규칙: 여기서 새 산출물을 만들지 않는다.** 검사하고, 고칠 데를 짚고, **작은 것만** 여기서 고친다. 큰 방향 문제는 되돌린다.
**스스로 대충 통과시키지 않는다.** 근거 없이 "좋아 보인다"로 넘기지 않는다 — 매 판정에 왜가 붙는다.

## 작동

### 0. 입력 읽기 + 준비 확인
- 4단계가 갱신한 `progress.md`의 **"생산"**(만든 것·생산물 위치)과 **"합격 기준"**(3단계가 정한 것)과 **"계획"**(구조·순서·유통·재사용·층)을 읽는다.
- **생산물이 없으면** 아직 안 만든 것 → drive-make(4단계)로 돌려보낸다. 여기서 즉석으로 만들지 않는다.
- **합격 기준이 없으면** 잴 자가 없는 것 → drive-plan(3단계)로 돌려보낸다. 여기서 즉석으로 기준 만들지 않는다 (잣대를 검사하는 사람이 만들면 자기 좋을 대로 만든다).
- `CLAUDE.md` 프로필 · `08-registry` (있으면 참고).

### 1. 검토자 자세 장착 (근거 없이 통과 금지)
이 단계의 값은 "깐깐하게 본다"에 있다. 그런데 마음가짐만으론 안 되고 **말투를 바꿔야** 작동한다.

**하지 말 것 (두루뭉술 통과 금지):** "잘 됐네요" · "괜찮아 보여요" · "이 정도면 충분해요" — 아무것도 검사하지 않는 맞장구. 전부 금지.

**대신 할 것:**
- **품질을 아는 전문가로 선다** — "돌아가나·됐나"만 보는 게 아니라 "제대로 됐나, 받는 사람에게 맞나, 대충 만든 티가 안 나나"를 본다.
- **대충 만든 티·뻔함에 무관용** — 이 산출물을 그 분야의 잘하는 전문가가 실제로 내놓을까? 그가 부끄러워할 뻔한 기본값(누구나 아는 흔한 틀·AI가 쓴 티가 나는 문구·속을 안 채운 겉치레)이 남아 있으면 짚는다. (4단계에서 피했어야 할 것이 실제로 안 남았는지 여기서 확인한다.)
- **비평은 구체적으로** — "느낌이 이상하다" 같은 막연한 말 금지. 네 가지로 적는다:
  - **관찰** — "여기 이게 보인다" (사실)
  - **질문** — "이건 받는 사람에게 어떻게 읽힐까" (짚어볼 것)
  - **제안** — "이렇게 바꾸면" (구체적 대안)
  - **근거 있는 의견** — "이게 약하다, 왜냐면 —" (판단 + 이유)
- **짚기 전에 끝까지 다 본다** — 산출물을 처음부터 끝까지 읽고 판정한다. 앞에서 안 보였다고 "빠졌다"로 단정하지 않는다 (뒤에서 다뤘을 수 있다). "이건 딴 데서 다뤘으니 됐다"고 넘길 땐 그 딴 데를 실제로 열어 확인한 뒤에만 — 확인 안 했으면 "미확인"으로 둔다.
- 좋은 점도 짚는다 — 검토는 흠만 찾는 게 아니라, 뭐가 섰고 뭐가 안 섰는지 가르는 것이다.

### 2. 축 A — 합격 기준 대조 (품질·느낌)
3단계가 정한 **합격 기준의 각 항목**을 산출물에 대고 하나씩 판정한다. ✓(섰다) / ✗(안 섰다) + **어디가 왜** 한 줄.
- **틀어지는 경우를 실제로 시험한다** — 3단계가 합격 기준에 "잘 될 때"뿐 아니라 "틀어지기 쉬운 경우"(분량 넘칠 때·톤 어긋날 때·핵심 빠질 때·빈 경우·처음 보는 사람)를 넣게 해뒀다. 여기서 그걸 **말로 넘기지 말고 실제로 넣어본다** — 긴 입력을 넣어보고, 처음 보는 사람 눈으로 읽어보고, 빈 경우를 만들어본다. 잘 될 때만 재면 5단계가 아무것도 못 거른다.
- **받는 사람 입장에서 읽는다** — 이 산출물을 대상(1단계가 정한 그 사람)이 받으면, 바라던 행동(읽고 신청? 보고 설득? 배우고 실행?)을 실제로 하게 되나. 만든 사람 눈이 아니라 받는 사람 눈으로 한 번 통과시킨다.
- **시선이 먼저 가는 곳** — 아무 설명 없이 처음 열었다 치고, 받는 사람 눈이 가장 먼저 닿는 두세 곳을 적어본다. 그게 내가 가장 보이고 싶던 핵심과 같나. 다르면 정작 핵심이 아닌 데로 눈이 먼저 가게 짜인 것이다 — 개별 항목이 다 합격이어도 전달은 빗나간다.
- **덜어낼 게 없나** — 요청 범위 안의 글에도 군더더기가 낀다. 자화자찬 도입부("저희는 최고의…")·아무도 안 읽는 긴 설명·같은 말 반복을 짚는다. "삼분의 일을 덜어내면 더 좋아진다" 싶으면 계속 덜어낸다. (3의 "덧붙음"과 다르다 — 그건 범위, 이건 밀도다.)
- **끝까지 한 호흡으로 훑어 마찰이 쌓이나 본다** — 항목별로는 다 통과해도, 처음부터 끝까지 읽을 때 잔말·불명확한 요청·과한 요구가 겹쳐 쌓이면 받는 사람을 잃는다. 마지막에 통째로 한 번 읽으며 이 누적을 본다.

### 3. 축 B — 계획 이행 대조 (완료 감사)
3단계가 약속한 것을 **목록으로 뽑아**, 생산물에 실제로 들어갔는지 하나씩 대조한다. (구조·순서 / 유통 경로 / 재사용하기로 한 것 / 층별 완성도 — AI 일은 풀버전으로 됐나, 사람 몫 판단은 반영됐나, 몸으로 할 일은 병목대로 잡혔나.)
- **계획 칸이 얇으면** 원 요청·데일리·초기 메모에서 약속을 끌어와 대조하되, "계획이 아니라 요청에서 유추 — 신뢰 낮음"으로 표시한다. 계획이 성기다고 그냥 통과시키면 과소 적발한다. 단 없는 계획을 검토자가 새로 지어내지는 않는다.

각 항목을 이렇게 판정한다:
- **완료** — 실제로 들어갔다 (어디에 있는지 근거를 댄다)
- **부분** — 시작은 했는데 덜 됐다 (뭐가 빠졌는지)
- **안 됨** — 약속했는데 없다
- **다르게 됨** — 계획과 다른 방식으로 갔지만 같은 목적을 이뤘다 (뭐가 달라졌는지). **너그럽게 본다** — 목적을 이뤘으면 지킨 것으로 치고, 방식이 계획과 다르다고 흠으로 깎지 않는다 (좋은 판단을 되돌리게 된다).
- **확인 불가** — 이 자리에서 있는지 없는지 확인이 안 된다 (사람이 무엇을 확인해야 하는지 적는다)

- **부분·안 됨은 왜인지도 짧게 본다** — 일부러 뺀 것(범위를 줄인 결정)인지 / 하다 만 것인지 / 뭔가에 막힌 것인지 / 잊은 것인지 / 요구를 잘못 이해해 빗나간 것(만들긴 했는데 겨눈 데가 다름)인지. 되돌릴지(4단계로) 받아들일지가 여기서 갈린다. 일부러 뺀 거면 되돌리면 안 되고, 잊었거나 빗나갔으면 되돌린다.
- **정직 규칙 (핵심)** — 관련된 걸 만들었다고 해서 완료로 치지 않는다. **"그 일을 다루는 것"과 "그 결과물"은 다르다** (예: 발행 도구를 붙였다고 글이 발행된 건 아니다). 애매하면 완료로 밀지 말고 **확인 불가**로 두고 사람이 확인하게 한다. 완료엔 인색하게, 확인 불가엔 정직하게. **단 확인 불가는 진짜로 이 자리에서 손이 안 닿는 것**(사람이 내릴 결정·외부 상태 대기)에만 쓴다 — 열어보면 알 수 있는 건 미루지 말고 실제로 확인해 완료/안 됨으로 판정한다 ("확인하기 번거롭다"는 손 안 닿음이 아니다). 안 그러면 정직 규칙이 "다 확인 불가로 던지고 아무것도 안 함"으로 변질된다.
- **덧붙은 것도 본다** — 약속한 게 다 들어갔나(빠짐)뿐 아니라, 계획·요청에 없던 절·기능·꾸밈이 산출물에 끼어들지 않았나(덧붙음)도 본다. 요청 안 한 덧붙임은 분량을 늘리고 초점을 흐린다 — 붙었으면 짚는다.
- **유통을 특히 본다** — 만들기만 하고 내보낼 경로가 실제로 연결됐나. 안 연결됐으면 아무도 못 본다. 단 **실제 발행·전달은 6단계(내보내기)** — 여기선 "경로가 연결됐나"만 확인한다.
- **여러 편·여러 절이면 편끼리 대본다** — 톤·부르는 말·용어·형식이 편마다 어긋나지 않나. 한 곳에 새로 넣은 것(항목·용어·값)은 그것을 나열하는 다른 자리(목차·요약·비교표·가격표)에도 다 반영됐나. 편을 각각 검토하는 것만으론 이 어긋남이 안 잡힌다.

### 4. 발견 정리 — 심각도 + 판정
축 A·B에서 나온 것을 정리한다.
- **발견마다 심각도** — 높음(첫인상·핵심·신뢰를 해침. 이대로 나가면 안 됨) / 중간(완성도를 깎음. 받는 사람이 은근히 느낌) / 다듬기(좋은 걸 더 낫게. 시간 되면).
- **겉이 매끈해도 근본 결함은 못 가린다** — 핵심 메시지가 없거나·무엇을 하라는지(요청)가 없거나·무엇인지 불명하면, 나머지가 아무리 매끈해도 높음이다. 문장이 아름다운 게, 디자인이 좋은 게 근본 결함을 통과시키지 못한다.
- **전체 판정 (셋 중 하나)**:
  - **합격** — 합격 기준을 서고 계획을 지켰다. 높음이 없고, 중간도 크게 안 걸린다.
  - **조건부** — 작은 것(중간·다듬기)만 고치면 통과. 큰 방향은 섰다.
  - **불합격** — 합격 기준의 핵심이 안 섰거나(높음), 계획이 약속한 핵심이 안 됐다. 되돌려야 한다.
- **근거 없는 판정 금지** — 각 판정에 왜가 붙는다.
- **깊이 우선** — 잘 짚은 대여섯 개가 모호한 스무 개보다 낫다. 사소한 걸 잔뜩 늘어놓아 큰 걸 묻지 않는다.
- **빠른 개선을 따로 뽑는다** — 효과가 큰데 금방 고칠 몇 개를 맨 앞에 모아준다 (먼저 고치면 티가 크게 나는 것).

### 5. 외부 시선 (독립 검토를 여러 각도로 — 큰 산출물만)
사람이 자기가 만든 걸 혼자 검토하면 놓친다. 한 번 훑어선 안 보이는 게 있다. **중요한 산출물이면** 독립 검토를 여러 각도로 병렬로 돌린다.
- 서브에이전트를 **서로 다른 각도**로 몇 개 띄운다 — 예: ① 받는 사람 입장(이게 나한테 와닿나) / ② 합격 기준 정합(기준을 진짜 서나) / ③ 냉정한 반대(이걸 실패시킬 구멍은 어디냐). 각자 발견을 반환하면 4의 정리에 합친다 (어느 각도에서 나왔는지 표시).
- **여러 각도가 같은 곳을 짚으면** 그건 진짜다 — 우선순위를 높인다.
- **작은 산출물**(캡션 하나 등)은 이 단계가 무겁다 → 생략하고 본인 검토로 끝낸다.

### 6. 처리 — 고치기 vs 되돌리기
판정에 따라 나눈다.
- **작은 문제(다듬기·중간 일부)** — 여기서 최소로 고치거나, 손볼 게 많으면 drive-make의 다듬기로 돌린다. 고친 것마다 **다시 대본다** (고쳐졌나 확인) + 분류: 고침 / 부분(더 손봐야) / 못 고침(되돌림).
- **큰 문제(불합격 · 높음)** — 여기서 억지로 때우지 않고 **되돌린다.** 생산이 부족하면 4단계(생산)로, 합격 기준이나 계획 자체가 잘못됐으면 3단계(계획)로. 어디로 왜 되돌리는지 적는다.
- **보류(제3의 칸)** — 진짜 문제지만 이 자리에서 손댈 수 없는 것(외부 자료·사람이 내릴 결정·아직 없는 정보 대기)은 고치지도 되돌리지도 말고 **"보류"로 표시해 다음 한 걸음으로 넘긴다.** 억지로 고치거나 그것 하나 때문에 전체를 되돌리지 않는다.
- **멈춤이 있다** — 무한히 고치지 않는다. 몇 바퀴 돌아도 안 끝나면 "더 고칠지 여기서 멈출지" 한 번 묻는다. 고치다 파장이 커지면(원래 문제와 무관한 데까지 손대게 되면) 멈추고 지금까지 한 걸 보여준 뒤 물어본다.

### 7. 최종 재검토
고친 뒤, 합격 기준으로 **한 번 더** 훑어 판정을 갱신한다.
- 고치다 오히려 나빠진 데는 없나(고친 게 다른 걸 망가뜨렸나) 확인한다.
- 최종 판정을 확정한다 (합격 / 조건부 → 남은 것 명시 / 불합격 → 되돌림).

### 8. 핸드오프 (산출)
`progress.md`를 갱신한다:
- **"검토"** 칸에 기록 — 전체 판정(합격/조건부/불합격) / 축 A 결과(합격 기준 항목별 ✓✗) / 축 B 결과(계획 이행 완료 감사) / 남은 확인 불가·되돌린 것 / 고친 것
- **"다음 한 걸음"** 갱신
- frontmatter `updated` 갱신. 통과·조건부면 `status: 리뷰`(조건부는 검토 칸에 남은 작은 수정을 함께 적는다), 되돌리면 `status: 생산` 또는 `계획`
- 로그에 `- <YYYY-MM-DD> 검토 <판정> (drive-review)` append
- 안내:
  - **합격**이면 — **"검토 통과했어요. 다음은 6단계 — 내보내기."**
  - **조건부**면 — 남은 것을 짚고, 고치면 통과라고 안내
  - **불합격**이면 — **"아직 안 섰어요. N단계로 되돌립니다."** 왜를 함께.

여러 산출물을 검토하는 프로젝트면(콘텐츠 여러 편 등) 산출물마다 이 스킬을 다시 돈다.

## 규칙 요약
- **만들기 금지** — 검사만. 작은 것만 여기서 고치고, 큰 구멍은 되돌린다.
- **근거 없이 통과 금지** — 두루뭉술 맞장구로 넘기지 않는다. 매 판정에 근거. 대충 만든 티·뻔함에 무관용.
- **두 축으로 본다** — ① 합격 기준 대조(품질·느낌) ② 계획 이행 대조(완료 감사). 하나만 보지 않는다.
- **틀어지는 경우를 실제로 시험한다** — 잘 될 때만 재지 않는다. 긴 입력·빈 경우·처음 보는 사람을 넣어본다.
- **정직 규칙** — 관련된 걸 만든 걸 완료로 치지 않는다. 애매하면 확인 불가로 두고 사람이 확인.
- **받는 사람 눈으로** — 만든 사람이 아니라 대상 입장에서 통과시킨다.
- **여러 각도로 본다** — 큰 산출물은 독립 검토를 병렬로. 여러 각도가 겹쳐 짚으면 진짜다.
- **판정은 셋** — 합격 / 조건부 / 불합격. 불합격은 되돌린다.
- **멈춤이 있다** — 무한 수정 금지. 안 끝나거나 파장 커지면 묻는다.
