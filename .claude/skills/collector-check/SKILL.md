# /collector-check — 수집기 검사 (검사표 6문 자동 실행)

비개발자가 "두 번 실행해서 증가 0 확인" 같은 검사를 직접 할 수는 없다. 이 스킬이 대신 실행하고 **통과/실패/사람확인**으로 보고한다.

## 읽는 파일

```bash
WS_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
```

- 선언표 `$WS_ROOT/00-system/08-registry/자리-선언표.yaml` 수집기 절 (대상 수집기의 선언)
- 기준 문서 `$WS_ROOT/00-system/03-guides/수집기-약속.md`

## 검사 절차 (6문)

대상 수집기마다:

**1문. 무중복 (약속 1)** — 실행명령이 있으면 **두 번 연달아 실행**하고 raw/ 파일 수를 전후 비교. 증가 0이면 통과. 실행명령이 없으면(손 수집) "받는 쪽이 고유 표식을 부여하는가"를 선언으로 확인.

**2문. 신선도 기록 (약속 2)** — 선언된 `상태파일`이 실존하고, 실행 후 마지막 시각이 갱신되는지 확인. 갈래가 여럿인데 기록이 하나뿐이면 "부분 열화 사각지대"로 지적. 상태파일 선언이 없으면 실패.

**3문. 충실도 (약속 3)** — raw 최신 표본 2~3건을 열어 잘림 신호(중간 끊김·말줄임·덮어쓰기 흔적)를 본다. 원천과의 글자 대조는 기계가 원천에 접근할 수 없으면 **사람확인**으로 넘긴다: "원본 앱에서 이 항목을 열어 끝부분이 같은지 봐 주세요."

**4문. 프라이버시 (약속 4)** — raw에서 OTP·인증번호 패턴(`인증번호|인증코드|verification` + 4~6자리)을 grep. 발견되면 "문 앞 차단(수집기 단 blocklist) 권고"로 실패 아님·경고.

**5문. 지속성 (약속 5)** — 선언의 `방식`·`알려진위험`에서 사람 돌봄 의존(로그인 갱신·앱 생존·기기 켜짐)을 읽어 나열. "며칠에 한 번 살려줘야 하나"를 명시.

**6문. 포착 (선언 1)** — 선언의 `포착`이 전량인지 골라 넣기인지 확인. 골라 넣기면 놓침을 잴 방법(주기 리마인드·실행신선도)이 선언돼 있는지 본다.

## 보고 형식

```
=== /collector-check: <수집기> ===
| 문 | 항목 | 결과 | 근거 |
|----|------|------|------|
| 1 | 무중복 | 통과/실패/해당없음 | 2회 실행 증가 0 |
| 2 | 신선도 기록 | 통과/실패 | 상태파일 갱신 확인 |
| 3 | 충실도 | 통과/사람확인 | 표본 N건 잘림 없음 |
| 4 | 프라이버시 | 통과/경고 | OTP 패턴 0건 |
| 5 | 지속성 | 정보 | 돌봄 의존: ... |
| 6 | 포착 | 통과/경고 | 전량 / 골라 넣기+감시 있음 |

판정: 붙여도 됨 / 보완 후 (□□) / 사람확인 대기 (3문)
```

- 실패·경고에는 **무엇을 고치면 되는지** 한 줄을 붙인다.
- 검사 결과 선언표와 실물이 다르면(상태파일 없음 등) **선언표 갱신을 먼저 권고** — 낡은 지도가 진실 행세를 하지 않게.
